<objective>
Implement the `runContractTests()` automated contract test suite (~25 checks) that validates any connector adapter against the full SDK contract, plus test helper factories (`createTestQuerySpec`, `createTestProfile`, `createTestSecrets`).

Purpose: This is the core deliverable of Phase 47 -- a reusable contract validation function that third-party plugin authors run against their adapters to catch >90% of common bugs (wrong response shape, missing entity extraction, broken pagination, auth failures) before publishing.

Output: `contract-tests.cjs` module with runContractTests and helpers, plus comprehensive unit tests proving all checks work against both valid and intentionally broken adapters.

function inferStatus({ events, entities, evidence, errors }) {
  if (errors.length > 0 && (events.length > 0 || entities.length > 0 || evidence.length > 0)) return 'partial';
  if (errors.length > 0) return 'error';
  if (events.length === 0 && entities.length === 0 && evidence.length === 0) return 'empty';
  return 'ok';
}

    **Main `runContractTests(createAdapter, options)` function:**

    Takes `createAdapter` (factory function returning an adapter object) and `options`:
    - `connectorId` (string, required) -- the connector ID
    - `mockResponses` (object, optional) -- maps `"METHOD /path"` patterns to `{ status, json }` fixtures. Default: single catch-all that returns `{ status: 200, json: { results: [{ id: '1', timestamp: new Date().toISOString(), name: 'test-event' }] } }`
    - `authConfig` (object, optional) -- `{ auth_type, secret_refs }`. Default: `{ auth_type: 'api_key', secret_refs: { api_key: { type: 'env', value: 'TEST_KEY' } } }`
    - `manifest` (object, optional) -- manifest for cross-check. If not provided, synthesizes from adapter capabilities.
    - `testEnv` (object, optional) -- env vars to set. Default: `{ TEST_KEY: 'test-value' }`

    Implementation uses `node:test` `describe` and `test` to create subtests. The function is `async` and should be called within a test context. Each check is a separate subtest with a descriptive name.

    **The ~25 contract checks (each a `test()` subtest):**

    Category 1 -- Structure (checks 1-3):
    1. `capabilities_valid` -- call `validateConnectorCapabilities(adapter.capabilities)`, assert valid=true
    2. `adapter_valid` -- call `validateConnectorAdapter(adapter)`, assert valid=true
    3. `manifest_cross_check` -- if manifest provided, verify adapter capabilities are superset of manifest auth_types, dataset_kinds, languages, pagination_modes

    Category 2 -- Preflight (check 4):
    4. `preflight_missing_profile` -- if adapter has `preflight`, call with null profile/secrets, assert it either throws or returns warnings (not silently succeeds with empty result)

    Category 3 -- Query Preparation (checks 5, 15):
    5. `prepare_query_produces_request` -- call `adapter.prepareQuery({ spec, profile, secrets, pagination, options })` where spec comes from `createTestQuerySpec`, assert result has `request` with `method` (string) and `url` (string)
    15. `idempotency` -- call prepareQuery twice with identical input, assert `JSON.stringify(result1) === JSON.stringify(result2)`

    Category 4 -- Execution (checks 6, 11):
    6. `execute_request_mock` -- start a `startJsonServer` using mockResponses, call adapter.executeRequest with the prepared request pointed at mockServer.baseUrl, assert no throw and response exists
    11. `timeout_handling` -- start a startJsonServer that delays 2000ms, create spec with `execution.timeout_ms = 100`, call executeRequest, assert it either throws a timeout error (code ETIMEDOUT) or the adapter handles it gracefully

    Category 5 -- Normalization (checks 7, 8, 12, 13):
    7. `normalize_response_events` -- call adapter.normalizeResponse with mock response, assert result has `events` array and `has_more` boolean
    8. `entity_extraction` -- if adapter.capabilities.supports_entities is true, assert normalizeResponse result includes `entities` array with at least one entry (or empty array is acceptable -- just must be present)
    12. `empty_response_handling` -- call normalizeResponse with empty response (`{ data: { results: [] } }` or similar), assert `events` is empty array and `has_more` is false
    13. `result_status_inference` -- use createResultEnvelope with various inputs to verify inferStatus logic: events-only -> 'ok', no events -> 'empty', errors+events -> 'partial', errors-only -> 'error'

    Category 6 -- Pagination (check 9):
    9. `pagination_cursor_propagation` -- if adapter supports cursor/token pagination, call normalizeResponse to get next_cursor, then call prepareQuery again with that cursor in pagination state, assert the prepared request differs from the first (URL or body includes cursor)

    Category 7 -- Error Handling (check 10):
    10. `error_propagation` -- start a startJsonServer returning 500, call executeRequest, assert the thrown error has `code`, `message`, and optionally `retryable` properties

    Category 8 -- Auth (check 14):
    14. `auth_header_attachment` -- start a startJsonServer that captures request headers, call `executeConnectorRequest` with the adapter's prepared request + profile + secrets, verify the response headers contain auth header for the configured auth_type (e.g., Authorization header for api_key/bearer/basic)

    Category 9 -- Optional Lifecycle (checks 16-21):
    16. `optional_preflight_callable` -- if adapter.preflight exists, assert it is a function and call completes
    17. `optional_emit_artifacts_callable` -- if adapter.emitArtifacts exists, assert it is a function
    18. `optional_on_error_callable` -- if adapter.onError exists, assert it is a function
    19. `lifecycle_stages_valid` -- if adapter.lifecycle is defined, assert every entry is in LIFECYCLE_STAGES
    20. `dry_run_support` -- if adapter.capabilities.supports_dry_run, call prepareQuery with `spec.execution.dry_run = true`, assert no throw
    21. `relationship_extraction` -- if adapter.capabilities.supports_relationships, assert normalizeResponse result includes `relationships` array

    Category 10 -- Integration shape (checks 22-25):
    22. `capabilities_has_required_fields` -- assert adapter.capabilities has id, display_name, auth_types (array), dataset_kinds (array), languages (array), pagination_modes (array)
    23. `prepare_query_request_shape` -- assert prepareQuery result.request has `method` that is one of GET/POST/PUT/PATCH/DELETE
    24. `normalize_response_shape_consistency` -- call normalizeResponse, assert result is plain object with events (array), has_more (boolean)
    25. `execute_request_returns_response` -- assert executeRequest returns an object with data property (or throws with code)

    Each subtest should be wrapped in try/catch so one failure does not abort the rest. Use `test()` from `node:test`.

    The function starts a single startJsonServer before all checks (using the mockResponses routing) and closes it after all checks. The mock server routes requests by matching `"METHOD /path"` against mockResponses keys, falling back to the default catch-all response.
